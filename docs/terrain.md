# terra-town — エリアの形と地形タイプ定義

> 本書は [Issue #3](https://github.com/rokusoudo-product/terra-town/issues/3)「エリアの形と種類の定義」の受け入れ基準を満たすための設計ドキュメント。

> 実装言語・ゲームエンジン・地図SDK（spec.md §11-8）が未確定のため、本書はコードではなく仕様定義として扱う。

## 1. マスの表現軸（地形軸 × 建築状態軸）

1マス（ヘクス）は **地形タイプ軸** と **建築状態軸** の直交する2軸で表現する。この2軸は独立して管理し、互いを兼ねない。

```
マス = (地形タイプ, 建築状態)
```

- **地形タイプ軸**: そのマスの土地としての性質（空き地・森・山・水辺・海・農地・市街）。産出資材を決める。
- **建築状態軸**: そのマスに建物が建っているかどうか（未建築 / 建築後）。プレイヤーの行動（建設）で変化する。

2軸を分離する理由:
- 「森のまま建物を建てる」ような矛盾を防ぎつつ、将来「地形タイプの変更」（例: 森を伐採して空き地にする）と「建築」を独立したアクションとして扱える。
- 産出ロジック（地形タイプ由来）と建築ロジック（建築状態由来）を疎結合にでき、`core/` 内でテストしやすい。

### 1.1 建築状態軸

| 状態 | 説明 |
|------|------|
| 未建築 | 建物が建っていない状態。地形タイプに応じた基礎産出のみ発生 |
| 建築後 | 建物が建っている状態。建物の種類・レベルに応じた効果が追加される（詳細は建物システムとして別途定義） |

- 建築状態を「建築後」にできるのは、地形タイプが **空き地** のマスのみ（後述 §2）。
- 建築後の地形タイプは引き続き「空き地」のまま扱う（＝建築は地形タイプを上書きしない。建物の存在は建築状態軸のみが表す）。

## 2. 地形タイプ一覧（7種）

| 地形タイプ | 産出資材 | 建築可否 | 備考 |
|-----------|---------|---------|------|
| 空き地 | なし（産出0、建築の土台） | 可 | 唯一、建物を建てられる地形。建築後は「空き地＋建築後」として扱う |
| 森 | 木 | 不可 | 伐採により空き地化する変更手段あり（§6） |
| 山 | 石・鉄 | 不可 | |
| 水辺（川・湖） | 水 | 不可 | |
| 海 | 塩 | 不可 | 建築不可だが開放対象（§7） |
| 農地 | 野菜・フルーツ | 不可 | |
| 市街 | 直接の産出なし（人口ボーナスに寄与） | 不可（＝プレイヤーが新規に建てるものではなく、OSM上の既存市街地として配置される想定） | §8 人口ボーナス参照 |

- 資材体系は `specs/001-mvp/spec.md` §6 の建設系（木・石・鉄）／生活系（塩・水・野菜・フルーツ）と対応する。
- 空き地は産出を持たないが、プレイヤーが建物を建てて発展させるための地形。産出ゼロの地形が偏ると詰みうるため、スポーン周辺の配置方針（§5）で担保する。
- 市街は「既に発展した街」を表す特殊地形で、プレイヤーが空き地に建てる建物とは別カテゴリ。実地図上の市街地（住宅密集地等）に対応させ、産出資材の代わりに人口ボーナス（§8）の母数になる。

## 3. ヘクスサイズ

- **1ヘクスの対辺（辺と向かい合う辺の距離）は約 50m** とする（仮置き、プロトタイプで調整）。
- 採用理由: 徒歩数十歩〜1分未満で隣接ヘクスへ移動できる粒度感を狙う。細かすぎるとGPS誤差（数m〜十数m）で地形判定がぶれ、粗すぎると「歩いた実感」が薄れるため、50m前後を初期値とする。
- 数値は `balance` 検討（plan/tasks工程）で最終調整する。変更する場合は本書を更新すること。

## 4. 内部判定はグリッド、UI表示のみヘクス（折衷方式）

2026-07-22 代表決定により、以下の折衷方式を採用する。

- **内部の開示判定・資材判定**: 緯度経度ベースの矩形グリッド（例: 数m四方の細分セル）に対して行う。GPS座標・OSMタグとの突き合わせは緯度経度のまま扱う方が実装・精度管理ともに単純なため。
- **UI表示**: プレイヤーに見せるマス目のみ、六角形（ヘクス）として描画する。デザイン性・拡張性（隣接6方向の演出のしやすさ等）を重視。
- **変換**: 複数の細分グリッドセルを1つのヘクスに集約し、セル群の地形タグの多数決（面積比が最大のタイプ）でヘクス全体の地形タイプを1つに確定する（§6 判定ルール参照）。開示判定（霧が晴れているか）もヘクス単位で集約する。
- この方式により、`core/`（判定ロジック）は緯度経度グリッドの矩形セル列を入力に取る形で実装でき、ヘクス座標系（表示用のみ）への依存を持たない。ヘクス⇔グリッドの変換は `location/` または表示レイヤーの責務とする。

## 5. OSMタグ → 地形タイプ判定ルール（初期版）

グリッドセル単位でOSMのタグと交差判定を行い、地形タイプを決定する。**複数タグが同一セルに重なる場合は、以下の優先順位（上が優先）で1つに確定する。** ヘクス単位への集約はセル多数決（§4）で行う。

| 優先度 | 地形タイプ | 判定タグ例（OSM） |
|--------|-----------|-------------------|
| 1（最優先） | 海 | `natural=coastline` の海側、`natural=bay`、`place=sea`、`natural=water` かつ `water=sea` |
| 2 | 水辺（川・湖） | `natural=water`（`water=lake` 等）、`waterway=river` / `waterway=stream` / `waterway=canal`、`natural=wetland` |
| 3 | 山 | `natural=peak`、`natural=mountain_range`、`natural=bare_rock`、`natural=scree`、`landuse=quarry` |
| 4 | 森 | `landuse=forest`、`natural=wood` |
| 5 | 農地 | `landuse=farmland`、`landuse=orchard`、`landuse=vineyard`、`landuse=meadow`（牧草地は農地扱い） |
| 6 | 市街 | `landuse=residential`、`landuse=commercial`、`landuse=industrial`、`building=*` が一定密度以上 |
| 7（フォールバック） | 空き地 | 上記いずれにも該当しないセル（`landuse=grass`、`landuse=greenfield`、`landuse=brownfield`、道路・未分類地含む） |

- 優先順位は「産出が特徴的で希少な地形」を上位に置く考え方（海・水辺・山は局所的に貴重、森・農地・市街は面的に広い、空き地は残余カテゴリ）。
- 本ルールは初期版であり、実データでの検証（誤判定・地域差）を経て plan/tasks 工程で精緻化する。
- 道路タグ（`highway=*`）は本Issueのスコープ外（§9参照）。現時点では地形判定に使わず、下地の地形タイプ（多くは空き地）に委ねる。

## 6. スポーン地点周辺の固定地形配置 と 地形タイプ変更手段

### 6.1 スポーン周辺の固定配置方針

- OSMの実データだけに依存すると、スポーン地点（初回起動位置）周辺が海・山などの単一地形に偏り、**産出ゼロで詰む**リスクがある。
- これを避けるため、スポーン地点から一定半径（目安: 数ヘクス〜十数ヘクス圏内、具体半径は plan/tasks で確定）については、以下を保証する。
  - スポーン地点そのもののヘクスは必ず **空き地**（＝プレイヤーが即座に建築を始められる）。
  - 半径内に、産出資材の主要カテゴリ（建設系・生活系それぞれ最低1種）を確保できる地形タイプが最低1ヘクスは存在する。
  - 実データ判定の結果、上記条件を満たさない場合は、**決定論的な補正**（スポーン地点の座標から一意に導かれるシード等を用いて、不足している地形タイプを近傍の空き地/低優先度セルに上書き配置する）で最低限の多様性を確保する。補正はプレイヤーごとに固定・再現可能とし、同じスポーン地点なら常に同じ配置になるようにする。
- 詳細な半径・補正アルゴリズムは plan/tasks 工程でバランス調整とあわせて確定する。

### 6.2 地形タイプ変更手段（概要）

- 森・山・水辺・農地などの非空き地は、初期状態では建築不可だが、**プレイヤーの行動によって地形タイプを空き地に変更できる**手段を用意する（例: 森を伐採して空き地化する）。
- 概要のみ本書で定義し、具体的な変更手段・必要資材・コストは plan/tasks 工程で設計する。
  - 想定: 対象地形に応じた行動（伐採＝森、開墾＝農地 等）と、行動に必要な資材コスト・実行条件（対象マスが開示済みであること等）を設ける。
  - 海・市街は変更対象外（海は建築不可のまま、市街はそもそもプレイヤーが変更する対象ではない）とする。

## 7. 海の開放・産出・ボーナスタイル方針

- **建築**: 不可（陸上の空き地のみが建築対象、§2）。
- **開放（霧を晴らす対象に含む）**: 海ヘクスも他の地形と同様に「開放対象」に含める。徒歩到達（実際に海沿いを歩いて隣接する等）でもポイント消費による開放でも、いずれの手段でも開放できる。
- **産出**: 開放済みの海ヘクスからは **塩** を産出する。
- **ボーナスタイル**: 海上には通常の海ヘクスとは別に「開放ボーナスタイル」（特殊オブジェクト）を一定間隔・条件で配置する。開放すると通常の塩産出に加えて特別な報酬（詳細は balance/tasks 工程で設計）を得られる。名所・観光POIオブジェクト（spec.md §3.1 V-B）の海版という位置づけ。
- 陸地に閉じない開放範囲にすることで、海岸沿いのユーザー体験を担保する（spec.md §11-2 開始位置ぼかしとも整合させ、海の開放自体はプライバシー方針の対象外＝個人の自宅特定に直結しないため制約なし）。

## 8. 人口ボーナスシステム（市街の産出ボーナス化）概要

市街は直接の資材を産出しない代わりに、**人口** を介して産出全体にボーナスをかける。

- **建物ごとの人口上限**: 建築後の建物（施設）は、それぞれ人口上限を持つ（例: 1施設あたり最大1000人、数値は仮）。
- **人口の増加**: 各施設の人口は時間経過で上限に向かって徐々に増加する（増加速度は balance 検討で確定）。
- **街全体の総人口**: プレイヤーの街に属する全施設の人口を合算したものを「総人口」とする。
- **産出ボーナス発動**: 総人口が一定の閾値（例: 10000人、数値は仮）に到達すると、**産出全体に倍率ボーナス**（例: x1.1）がかかる。閾値は複数段階（人口が増えるほどボーナスが上がる）にすることも plan/tasks で検討する。
- 市街（OSM実地図の市街地由来の特殊地形）自体が人口ボーナスの母数に含まれるのか、プレイヤーが建てた建物のみが対象なのかは、plan/tasks 工程でゲームバランスと合わせて確定する（初期案: プレイヤーが建てた建物の人口を主対象とし、市街ヘクスの近接はボーナス倍率の補正要因とする）。
- 具体的な成長速度・閾値テーブル・倍率カーブは balance 検討（plan/tasks 工程）で確定する。本書では仕組みの概要のみを定義する。

## 9. スコープ外（本Issueでは扱わない）

- **道の情報**（OSM `highway=*` 等を用いたリアルタイム経路表現・道沿いボーナス等）は MVPスコープ外。将来Issueで再検討する。
- 具体的な数値（人口閾値・伐採コスト・スポーン補正半径・ヘクスサイズの最終値等）は balance 検討として plan/tasks 工程で確定する。
- 建物の種類・レベル・強化ロジックの詳細（建築状態軸の「建築後」を更に細分化するかどうか）は別途、建物システムとして定義する。
