# terra-town — 開放ポイントの定義

> 本書は [Issue #4](https://github.com/rokusoudo-product/terra-town/issues/4)「開放ポイントについて」の受け入れ基準を満たすための設計ドキュメント。
> `specs/001-mvp/spec.md` §3.2（歩く動機の担保）§5（マップ設計方針）§6（ゲーム内経済）§9 NFR-2（位置情報プライバシー）を具体化する。
> 実装言語・ゲームエンジン・地図SDK（spec.md §11-8）が未確定のため、本書はコードではなく仕様定義として扱う。
> 実装フェーズでは `C:\Users\moets\.claude\GPS_ARCHITECTURE.md`（`core/` は `location/` を import しない一方向依存）に従うこと。
> **開放ポイントの経済ロジック（入手量計算・ストック管理・開放コスト判定・隣接制約判定）は `core/` 相当の純粋ロジックとして実装する。GPS移動距離の取得・歩数センサーとの突合・立入禁止エリア判定用の地理データ取得は `location/` 相当のモジュールに置き、`core/` へは「距離(m)」「経過日数」等の計算済みの値のみを渡す。**

## 1. 背景・目的

- 歩くだけではどうしても開放できないポイントが存在する（立ち入り禁止エリア・海など）。
- ポイント（以下「開放ポイント」）を消費することで、歩く以外の手段でも未踏破エリアを開放できるようにする。
- ただし `docs/terrain.md` §5〜§6 の「霧の開放は歩行が基本」という設計、および spec.md §3.2「歩く方が明確に有利」という方針は維持する。開放ポイントはあくまで**救済・補完手段**であり、歩行を代替するほど強い手段にはしない。

## 2. 開放ポイントの入手（自然回復＋歩行距離換算）

開放ポイントは以下の2系統の合算で入手する。

### 2.1 自然回復（1日1P・固定）

- **1日ごとに固定で 1P** を自然回復として付与する。
- 2026-07-22 代表回答により、Issue本文の当初案（1時間で1P＝24P/日相当）から**大幅減量**して確定した。理由: 歩行によるポイント獲得を主軸とし、自然回復だけで潤沢にポイントが貯まる設計を避けるため（歩行優位の担保）。

### 2.2 歩行距離換算（GPS移動距離ベース）

- 歩いた distance（GPS移動距離の累積）に応じて開放ポイントを付与する。
- **換算レート（暫定）: 1.5km = 1P**
  - ⚠️ **本レートは secretary 提案の仮置き値であり、代表の最終確認待ちである。** plan/tasks 工程、または balance 検討で正式に確定・変更されうる。
  - 参考シミュレーション（暫定レートの場合）: 1日 7km 歩行 → 約 4.6P。自然回復 1P と合算すると **1日あたり約 5.6P 相当**となり、自然回復のみ（1P/日）と比べて歩行側が明確に優位になる水準を狙っている。
- 距離の丸め・切り捨てタイミング（例: 1.5km到達ごとに都度加算するか、日次で合算してから換算するか）は plan/tasks 工程で確定する。

### 2.3 入手源のまとめ

| 入手源 | レート | 備考 |
|--------|--------|------|
| 自然回復 | 1P / 日（固定） | 2026-07-22 代表決定。上限到達後は無駄になる（§4） |
| 歩行距離換算 | 1.5km = 1P（暫定・要最終確認） | GPS移動距離が一次ソース（§3） |

## 3. 一次ソース: GPS移動距離、歩数センサーとの役割分担

- 開放ポイントの歩行距離換算（§2.2）は **GPS移動距離を一次ソース**とする。
- **歩数センサー（step counter）は、開放ポイント付与の直接の入力にはしない。** 役割は以下に限定する。
  - GPS位置偽装対策（spec.md §9 NFR-3、`docs/concept.md` の「GPS位置偽装対策の候補」参照）としての**突合用**。歩数と移動距離が著しく乖離する場合（例: 歩数がほぼゼロなのに長距離移動している）、開放ポイント付与・エリア開示の資材獲得レートを抑制する不正対策シグナルとして利用する。
  - `docs/terrain.md` が扱う「地形/産出」の判定ロジックとは独立した論点であり、本書では歩数センサーの具体的な閾値・アルゴリズムまでは定義しない（plan/tasks 工程で確定）。
- この役割分担により、`core/`（開放ポイント経済ロジック）が受け取る入力は「GPS移動距離（+ 偽装対策済みフラグ）」のみとし、歩数データの生値は `location/` 側で処理してから core へ渡す。

## 4. ストック上限（50）

- 開放ポイントは**最大 50 ポイントまでストック可能**とする。
- 上限に達している間は、自然回復（§2.1）・歩行距離換算（§2.2）による新規付与分は加算されずに切り捨てられる（貯めすぎ・非稼働放置による無限蓄積を防ぐ）。
- 上限値の妥当性（50が多すぎ/少なすぎないか）は、開放コスト（§5）1メッシュ=1ptと合わせて balance 検討（plan/tasks 工程）で再検証しうる。

## 5. 開放コスト（1pt/メッシュ）と隣接制約

### 5.1 開放コスト

- 未踏破メッシュ（`docs/terrain.md` の言うヘクス単位。内部判定はグリッド、UI表示はヘクスの折衷方式に準拠）を開放ポイントで開くコストは、**地形タイプに関わらず一律 1pt/メッシュ（固定）** とする。
- 2026-07-22 代表回答により、当初検討していた「遠距離ほどコストが逓増する」案は**不要**と確定した。理由: 下記の隣接制約（§5.2）により、遠隔地の飛び地開放は事実上防止できるため、距離に応じたコスト逓増を別途設ける必要がない。

### 5.2 隣接制約（地続き拡張のみ）

- 開放ポイントで開放できるのは、**既に開放済みのメッシュに隣接するメッシュのみ**とする。
- 開放済みメッシュに隣接しない、飛び地状態のメッシュはポイントで開放できない（=遠隔地へのワープ的な開放は不可）。
- これにより「歩いて到達した領域から地続きに広げる」という体験を保ちつつ、ポイントは"あと一歩"のギャップを埋める救済手段として機能する。
- 隣接の定義は `docs/terrain.md` §4 のヘクス表示（UI）に準拠し、UI上で隣接する6方向のヘクスを対象とする（内部判定は同節の折衷方式に従い、グリッド集約後のヘクス単位で判定する）。

## 6. 開放対象範囲（海 と 実在の立入禁止エリアの区別）

開放ポイントによる開放の可否は、地形タイプごとに以下のとおり区別する。

| 対象 | 開放ポイントでの開放 | 産出 | オブジェクト配置 | 備考 |
|------|----------------------|------|------------------|------|
| 海 | **可** | 開放後は塩を産出（`docs/terrain.md` §7 準拠） | 通常のオブジェクト配置あり（海版の名所・ボーナスタイル含む） | コンプリート志向プレイヤー向けの到達目標として位置づける |
| 実在の立入禁止エリア（軍事施設・私有地・危険区域等） | **不可（対象外）** | なし | なし | ゲーム上は**黒塗り（マスク表示）**にする |
| 上記以外の通常メッシュ | 可（§5の条件下） | 地形タイプに応じた通常産出 | 通常のオブジェクト配置あり | |

- **海**: `docs/terrain.md` の方針と整合させ、開放ポイントでの開放を許可する。歩行での到達（海沿いを歩いて隣接させる等）と、開放ポイントの両方の手段を認める。
- **実在の立入禁止エリア**: プレイヤーを現実の立入禁止区域へ誘導しないための安全上の配慮として、ゲーム上は黒塗り（マスク表示）にし、開放不可・産出なし・オブジェクト配置なしとする。spec.md §9 NFR-2（位置情報プライバシー・取り扱い方針）と同じ思想に基づく制約。
  - どの区域を「立入禁止」と判定するかのデータソース（例: 国土数値情報の指定区域データ、OSMの `access=no` / `military=*` 等のタグ等）は本書のスコープ外とし、**plan 工程で確定する**。
  - データソース確定までの暫定運用（例: 判定できない区域はひとまず通常メッシュとして扱うか、安全側に倒して非表示候補としてリストアップするか）も plan 工程で決める。

## 7. V-B（土地の固有性）実現方針の変更との関係（参考・本Issueのスコープ外だが関連するため付記）

- 2026-07-22 代表回答により、spec.md §3.1 V-B・§3.2 が前提としていた「固有名所は現地限定」という制約は**採用しない**方針に変更された。理由: 現地限定にすると立入禁止・遠隔地等、行けない場所の方が多くなり離脱要因になるため。
- 代わりに、「行ってみたい」「ここにはこんなものがある」と興味を引くオブジェクトを地図上に大量配置する方向でV-Bを実現する（希少な現地限定コレクションではなく、豊富な発見コンテンツで動機づけ）。
- この方針転換は本Issue（#4 開放ポイント）の決定事項に含まれるが、内容としては spec.md §3.1/§3.2 の記述に影響する。本書では開放ポイントの仕様定義の一部として記録するに留め、**spec.md 本文自体の改訂は別途行う**（本Issueのスコープ外。次回 spec.md 更新時に反映する）。

## 8. スコープ外（本Issueでは扱わない）

- 換算レート「1.5km = 1P」の最終確定（代表確認待ち。§2.2）。
- ストック上限50の最終バランス検証（§4）。
- 実在の立入禁止エリア判定に使う具体的なデータソース・アルゴリズム（§6）。
- 隣接制約の判定を含む具体的な実装（座標系・データ構造）は、ゲームエンジン確定後に plan/tasks 工程で設計する。
- 歩数センサー突合の具体的な閾値・不正検知アルゴリズム（§3）。
