# 仕様書: map_game — 歩いて地図を開示し、資材で街を育てる位置情報ゲーム

## 概要

GPS で実際に歩いた場所の地図が開示されていく(霧が晴れる)ゲーム。開示済みエリアからは資材が獲得でき、資材を使って自分の街を育てる。歩いていない場所もポイント消費で開示可能。「探索(リアル移動)」と「育成(街づくり)」の2つのループを組み合わせる。

**方向性決定: 2026-07-08 代表回答**(当初の案B「動く」+ 案C「育てる」のハイブリッド)

**コンセプト確定: 2026-07-20 代表回答** — 本作の特徴(価値提案)を以下に定める:

- GPS 機能を用いてマップを歩く。**歩いた場所が開拓され、自分の街が育っていく**
- **歩く楽しさ + 健康促進 + マップの開拓** を組み合わせる
- 行ったことがない場所も**ポイントで開拓**できる
- → 最終的に「**地球が自分の街になる**」楽しさを目指す

## コアゲームループ

```
歩く(GPS)
  → 通った場所の霧が晴れて地図が開示される
  → 開示エリアから資材を獲得(エリアの特性で資材が変わる想定)
  → 資材で自分の街を建設・拡張
  → 街の成長でポイント獲得
  → ポイントで「歩いたことのない場所」も開示できる
  → 開示地からさらに資材 → …
```

- **リアル行動の報酬化**: 日常の移動・散歩がそのままゲーム進行になる
- **ポイント開示**: 遠隔地・行けない場所もポイントで開ける救済/ショートカット。歩く方が効率的なバランスにして「歩く動機」は保つ

## 機能要件(初期整理)

### 探索系

- GPS で移動経路を記録し、通過したエリア(メッシュ/タイル単位)を開示済みにする
- 未開示エリアは霧(fog of war)表示
- ポイント消費による任意エリアの開示
- 開示済みエリアからの資材獲得(例: 水辺→水、山→木材/鉱石、市街地→部品、など地図の地物と連動)
- **特殊マス**: 畑・鉱山・森などの資材産出地や観光地は、開示すると特別なオブジェクトが表示される。実在の土地利用データ(OSM の farmland / forest / quarry、観光 POI 等)と連動させ、「現実にそこにあるもの」がゲーム内の産出地になる

### 育成系(2026-07-08 代表回答で方向決定)

- **街は実在地図の上に建てる**。開示済み(霧が晴れた)エリアに建物を配置していく
- エリアは**自治体(市区町村)の区画**をベースに区切り、それぞれ異なる特徴・役割を持たせる(自治体ごとに産出資材や発展ボーナスが変わる想定。行政区域データは国土数値情報等の無償ソースを利用可能)
- 獲得資材で開示エリア内に建物を建設・アップグレード
- エリア(自治体)ごとの発展度に応じてポイント・資材獲得効率が向上(探索で得る資材とエリア特性の相性を作る)
- 「自分が実際に歩いた土地に自分の街が広がっていく」体験が本作の核

### 共通

- オフライン時も移動記録を貯め、オンライン復帰時に反映

## 非機能要件(初期整理)

- バッテリー消費を抑えた位置記録(常時高精度 GPS は使わない)
- 位置情報のプライバシー保護(経路データの扱い・利用規約)— 位置ゲーの必須要件
- 地図データ: OpenStreetMap / 国土地理院タイル等の無償ソースを想定

## MVP(最小構成)の提案

まず「探索ループだけ」を成立させる:

1. 歩いた場所の霧が晴れる地図(これだけでも Fog of World 系として遊べる)
2. 開示エリア数に応じた資材カウント(獲得ロジックは単純に)
3. 街育成は最初は簡易(建物数種類・レベル制)

→ 探索の気持ちよさを検証してから育成を深める。

## 追加決定事項(2026-07-20 代表回答・第2回)

- **公開範囲: 一般公開前提**。これに伴い位置履歴の保存先・プライバシー方針・バックエンドの要否を spec/plan で具体化する。特に**開始時の位置(自宅等)をどうぼかすか**を検討する(候補は下記)
- **GPS 位置偽装対策: 必要**(一般公開のため。候補は下記)
- **資材の体系**(種類数はバランスを見て決定):
  - 建設系: **木・石・鉄** → 建物などの建設に使う
  - 生活系: **塩・水・野菜・フルーツ** → 人(住民)の効率 UP に使う
- **エリア特性の決め方**:
  1. 有名な観光地などのエリアは**オブジェクトとしてボーナス**を持つ(例: 東京 → 東京タワー・皇居)
  2. エリア特性は**実際の地図のその場所の特性を反映**する(例: 森の近くは木が取りやすい、砂漠の辺りは石や鉄が取りやすい)

### 開始位置のぼかし方の候補(2026-07-20 secretary 提案。spec.md で選定)

1. **公開レイヤーの分離(推奨・最も安全)**: 歩行経路・開拓メッシュは**本人のみ閲覧可**とし、他人に公開するのは「自治体単位の発展度」「育てた街のビュー」だけにする。経路データ自体を共有しない設計なら、ぼかしの精度議論そのものが不要になる
2. **プライバシーゾーン(Strava 型)**: 自宅など指定地点の半径 200m〜1km を公開データから除外。初回起動地点の周辺を**デフォルトで自動ゾーン化**すると設定漏れを防げる
3. **端点トリム**: セッションの開始・終了付近の数百 m を公開データから自動カットする
4. **精度の丸め**: 公開・サーバー保存する位置をジオハッシュ等で粗い精度(例: 数百 m メッシュ)に丸める

→ 推奨の組み合わせ: **1 を基本設計にし、2 を保険として併用**(将来、経路共有機能を追加する場合に 3・4 を導入)

### GPS 位置偽装対策の候補(2026-07-20 secretary 提案。spec.md で選定)

1. **モック位置検出**: Android の `Location.isMock`(模擬位置フラグ)と模擬位置アプリの検出。低コストで基本対策になる
2. **速度・テレポート検出**: 前回位置から物理的に不可能な移動(例: 徒歩相当の時速上限超え、瞬間移動)は開拓・資材付与を無効化する。車移動は開拓レートを下げる等の調整も可能
3. **歩数センサー突合**: 歩数計(step counter)と移動距離を突き合わせ、「歩いていないのに移動している」場合は資材獲得レートを落とす
4. **Play Integrity API**: root 化端末・エミュレータ・改造アプリの検出。一般公開・ランキング導入時には必須級
5. **サーバー側の異常検知**: 位置履歴の統計的な異常パターン検出(将来。バックエンド実装後)

→ 推奨の MVP セット: **1 + 2 + 3**(クライアント側で完結し実装コストが低い)。一般公開リリース時に **4** を追加、ランキング等の競争要素を入れる段階で **5** を検討
