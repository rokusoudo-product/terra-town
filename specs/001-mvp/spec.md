## 1. 概要

GPS で実際に歩いた場所の実地図が開示（霧が晴れる）され、**現実の土地の特性がそのまま資材・名所になり、歩いた土地の上に自分の街を育てる**位置情報ゲーム。
「探索（リアル移動）」と「育成（街づくり）」の2ループを、**実在地図との連動**で結びつける。最終ビジョンは「**地球が自分の街になる**」。

- プラットフォーム: **Android 先行**（iOS 展開も見据える）。
- プロジェクト名: **terra-town**（確定）。

## 2. 背景・目的

- 日常の移動・散歩をゲーム進行に変え、**歩く楽しさ＋健康促進＋マップの開拓**を組み合わせる。
- 位置情報ゲームの資産（GPS/地図/測位まわり）を今後の GPS 利用アプリで再利用できる形に育てる（GPS_ARCHITECTURE 準拠、§10 参照）。

## 3. 価値提案の再設計 ★本 spec の中心（要代表判断）

フィードバック「地図ベースに寄せる／GPSの利点を明確に」に応え、**"GPS/実地図でなければ成立しない面白さ"** を核に据える。
均質グリッド（＝どこを歩いても同じ）が失敗の元だったため、**「その場所ならでは」を価値の中心**にする。

### 3.1 GPS の価値の柱（候補・組み合わせる）

- **V-A 実地図連動＝現実の再発見**: 開示されるのは抽象タイルでなく**実在の地図**。歩いた通り・公園・川・駅がそのまま自分の街の一部になる。「自分が歩いた現実」が可視化・所有される感覚が核。
- **V-B 土地の固有性（場所ならでは）**: 実在の地物・土地利用（森=木、山=石/鉄、水辺=水、市街=部品、農地=作物）と、**観光地/名所は固有オブジェクト**（例: 東京タワー・皇居）として出現。**その場所に行かないと手に入らない**資材・名所コレクションを作る。
- **V-C 自治体区画の意味づけ**: エリアを市区町村区画で区切り、区画ごとに特性・発展ボーナス・「制覇率」を持たせる。**行政区画を集める/育てる**という地図ならではの目標。
- **V-D 現実連動の変化（任意）**: 歩数・時間帯・季節・（将来）天候で産出や見え方が変わる。健康促進と接続。

> **secretary 推奨のMVP軸**: **V-A＋V-B を核**（実地図の上で、その土地ならではの資材・名所が手に入る）。V-C は「区画制覇」という中期目標として MVP に薄く導入。V-D は将来。
> ⇒ この価値の柱の採否・優先度が **最重要のゲート①判断事項（§11-1）**。

### 3.2 「歩く動機」の担保

- ポイント開示（未訪問地をポイントで開く救済）は残すが、**歩く方が明確に有利**（範囲開示・固有名所は現地限定・コスト差）にして、GPSの意味を守る。

## 4. コアゲームループ（GDD の核）

```
歩く（GPS で移動を記録）
  → 通った場所の霧が晴れ「実地図」が開示される（V-A）
  → 開示エリアの“その土地の特性”に応じた資材／固有名所を獲得（V-B）
  → 資材で開示エリア（実地図上）に建物を建設・強化 → 街が育つ
  → 街の発展・区画制覇でポイント獲得（V-C）
  → ポイントで遠隔地も開示（救済）。ただし歩く方が有利
  → さらに歩いて開拓 → …
```

## 5. マップ（レベル）設計方針

- **舞台は実世界の地図全体**。単位は「開示メッシュ（タイル）」＋「自治体区画」の2層。
- メッシュ: 歩行で開示する最小単位。地物タイプで産出資材が決まる。
- 区画（市区町村）: 複数メッシュの集合。制覇率・発展度・エリアボーナスを持つ。
- **特殊マス/名所**: 実在の土地利用・観光POI と連動した固有オブジェクト（現地限定）。
- データ源: OpenStreetMap / 国土地理院タイル / 国土数値情報（行政区域）等の無償ソースを想定（plan で確定）。

## 6. ゲーム内経済（資源の入手と消費）

- **建設系資材**: 木・石・鉄（建物の建設・強化に使用）。
- **生活系資材**: 塩・水・野菜・フルーツ（住民/街の効率UPに使用）。
- **入手**: 開示メッシュの地物特性に連動（森→木、山→石/鉄、水辺→水、農地→作物、市街→部品 等）。
- **ポイント**: 街の発展・区画制覇で獲得。未訪問地の開示に消費（歩行優位のバランスを維持）。
- 資材の**種類数は絞るか全採用か**を §11 で確定（数値正本は将来 `balance.csv` で管理）。

## 7. 進行曲線・クリア条件

- 明確な「終わり」はない育成型。中期目標として **区画制覇率・街レベル・名所コレクション達成率** を段階目標に。
- 進行の牽引: 「隣の区画を制覇したい」「あの名所を取りに行きたい」という**地図上の到達欲**。

## 8. ユーザーストーリー

- **US-1（散歩ユーザー）**: 通勤・散歩で歩いた場所の実地図が開拓され、自分の街が広がるのを見たい。
- **US-2（収集ユーザー）**: その土地でしか取れない資材・名所を集めたい（現地に行く動機）。
- **US-3（育成ユーザー）**: 集めた資材で実地図上に街を建て、区画を制覇して育てたい。
- **US-4（プライバシー懸念ユーザー）**: 自宅など開始位置が他人に分からない形で安心して遊びたい。

## 9. 機能要件 / 非機能要件 / 受け入れ基準 / スコープ外

### 機能要件（MVP）
- FR-1 GPS 移動を記録し、通過メッシュを開示（霧を晴らす）。**実地図タイル上**に表示。
- FR-2 開示メッシュの地物特性に応じて資材を付与。
- FR-3 実在の名所/土地利用と連動した**固有オブジェクト**を現地開示時に出現させる。
- FR-4 開示エリアに建物を建設・強化できる（実地図上）。
- FR-5 自治体区画の制覇率・発展度を集計・表示する。
- FR-6 ポイントで未訪問メッシュを開示できる（歩行優位のコスト設計）。
- FR-7 オフライン移動を蓄積し、オンライン復帰時に反映。
- FR-8 開始位置ぼかし等の位置プライバシー保護を行う（§11・plan で方式確定）。

### 非機能要件
- NFR-1 バッテリー消費を抑えた位置記録（常時高精度GPSを使わない）。
- NFR-2 位置情報プライバシー保護（経路データの扱い・利用規約）。位置ゲーの必須要件。
- NFR-3 **GPS位置偽装対策**（一般公開前提。§11・plan で採用セット確定）。
- NFR-4 無償地図データソースのライセンス順守（表記含む）。

### 受け入れ基準
- [ ] §11 判断事項（特に価値提案の柱）が代表レビューで解決される。
- [ ] MVP スコープ（実地図連動を核とする探索→育成ループ）が承認される。

### スコープ外（当面）

- 課金・収益化。
- 天候/季節連動（V-D）等の現実連動の高度化。

## 10. 設計上の制約（GPS_ARCHITECTURE 準拠・plan で具体化）

- ゲームロジックの純粋実装（`core/`：開示判定・資材・建設・経済）は、GPS/地図（`location/`）を **import しない**。依存は `location/ → core/` の一方向。
- 目的: GPS/地図まわりを今後の位置情報アプリで再利用できる資産にする。
- プロトタイプでも「探索/開示ロジック」と「移動入力（疑似移動）」を関数レベルで分離済み。
